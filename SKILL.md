# document-system（資料化システム）

相談・報告内容をパワーポイント資料（.pptx）に変換する。テキストによる状況説明・Excel/CSV等の数値データの両方を受け付け、内容に応じて適切な図解タイプを自動判断し、統一デザインのスライドを生成する。

> 詳細なルール・実装パターンは同ディレクトリの `AGENT.md`・`design-system.md`・`skills/*.md` に定義されている。このSKILL.mdはそれらの核心部分を含んでおり、**このファイルだけでも基本動作できる**が、実装の精度を上げるために必ず関連ファイルも読むこと。

## 起動時に必ず読むもの

1. **`design-system.md`**（同ディレクトリ）：色・フォント・余白の唯一の基準
2. **`AGENT.md`**（同ディレクトリ）：全体フローの詳細ルール
3. **`skills/` 配下の該当する図解skill**：後述の判断ロジックで選定したものだけを読む

## 全体フロー

### STEP 0：ヒアリング（対話形式・寄り添う姿勢で）

ヒアリングの質がスライドの質の9割を決める。4つのフェーズで段階的に聞く。質問攻めにせず、1回に1〜2問。受け止めの一言を挟んでから次に進む。ボタン選択式は使わない。

**フェーズ1：入口** — 「何を相談しますか？」から入る。内容がまとまっていそうなら貼り付けを促す。漠然としていれば対話で引き出す。

**フェーズ2：核心の3問**（これが埋まるまで構成に進まない）
- **① ゴール**：「この資料を見た相手に、最終的にどう動いてほしいですか？」（承認／判断／理解／予算／協力…）
- **② 状況**：「今どういう状況ですか？」（課題・背景・経緯・関連する数値）
- **③ 緊急度**：「なぜ今のタイミングで伝える必要がありますか？」（期限・悪化・意思決定の時期）

**フェーズ3：説得力を上げる深掘り**（内容に応じて必要なものだけ）
- **④ 相手の前提知識**：初めて聞く話か、ある程度知っているか → 背景説明の厚さが変わる
- **⑤ 相手が反対・懸念しそうな点**：コスト？期間？リスク？ → 比較表や根拠で先回りして潰す
- **⑥ 選択肢・比較対象**：他の案があるか → あれば比較表、なければ提案1本の構成にする
- **⑦ 数値データの有無**：Excel/CSV/メモ → あれば貼り付けを促す

**フェーズ4：認識合わせ** — 得た情報を「■相手 ■ゴール ■課題 ■なぜ今か ■前提知識 ■懸念 ■データ有無」で要約し、合っているか確認。OKならSTEP1へ。

**姿勢**：受け止めてから次へ／言語化を手伝う／急かさない

### STEP 1：構成案の設計（図解タイプの自動判断）

内容を要素分解し、下記の対応表で各スライドの図解タイプを決める。3つ以上の箇条書きになりそうな要素は原則として図解に変換する。構成案はユーザーに一度提示して認識合わせする。

| 伝えたい内容 | 図解タイプ | 参照するskill |
|---|---|---|
| 手順・原因→結果の流れ、承認/処理の経路 | フロー図 | `skills/flow-diagram.md` |
| 数量の比較・推移・構成比などの数値データ | グラフ | `skills/chart-graph.md` |
| 組織・体制・役割分担、要素同士の関係性 | 体制図・関係図 | `skills/org-chart.md` |
| 複数の案・選択肢の比較検討 | 比較表 | `skills/comparison-table.md` |
| スケジュール・時系列の推移 | タイムライン | `skills/timeline.md` |
| 2軸で分類・評価（優先度、緊急度×重要度など） | マトリクス図 | `skills/matrix-diagram.md` |
| 中長期の計画・フェーズ展開 | ロードマップ | `skills/roadmap.md` |

1スライドに図解は1つが原則。迷うときはユーザーに聞く。

### STEP 2：HTMLでスライドを組む（見た目確定）

`deck.html` を1ファイルで作成。各スライドを `<section class="slide">` として縦に並べる。

- **まず `templates/reference-deck.html` を読む**。これがA案デザインの完成形見本であり、CSS・レイアウト・各図解の「正解」。新しいスライドはこれを踏襲して作る
- 1280×720px固定（16:9）、ブラウザで開けばレビュー可能な状態にする
- 各スライドの図解は、選定した `skills/*.md` の実装パターンに従う
- テキストのみのスライド（視覚要素が何もない）は作らない

### STEP 3：HTMLを参照してpptxへ変換

`python-pptx` を使い `deck.pptx` を生成。HTMLの内容・レイアウト・配色を忠実に再現する。

### STEP 4：変換前チェックリスト

- [ ] ネイビー・白・グレーのみ（`design-system.md` 以外の色を使っていない）
- [ ] フォントがメイリオで統一されている
- [ ] 全スライドタイトルが結論文になっている
- [ ] 2枚目がサマリー（結論＋してほしいこと）になっている
- [ ] 3つ以上の箇条書きが並ぶスライドがない（図解化されている）
- [ ] 装飾・グラデーション・影・3Dを使っていない

## デザインルール要約（詳細は design-system.md）

**色**：`--navy: #1B2436` / `--white: #FFFFFF` / `--gray-lt: #F2F3F5` / `--gray-border: #D8DAE0` / `--gray-mid: #9AA3B5` の5色のみ。アクセントカラーは一切使わない。強調は濃淡と太字で行う。

**フォント**：メイリオ固定。pptx変換時も `font.name = "Meiryo"` を全テキストに明示指定。

**レイアウト**：16:9（13.333×7.5インチ）。表紙・最終スライドのみネイビー背景＋白文字。中間スライドは白背景。タイトルは結論文にする。

---

## 推奨スライド構成（内容に応じて取捨選択）

1. 表紙（ネイビー背景）
2. **サマリー**（結論＋してほしいこと）← 2枚目に必ず置く
3. 背景・経緯
4. 現状・課題
5. 原因分析
6. 提案・解決策
7. 比較・選択肢
8. スケジュール・体制
9. 次のアクション（ネイビー背景）

---

## STEP 5：納品後のテンプレート更新確認

pptx納品後、ユーザーが出来栄えに満足している様子であれば：
> 「今回の資料、仕上がりが良さそうですね。今後のお手本として、テンプレートに保存しておきましょうか？」

「はい」→ `deck.html` を `templates/reference-deck.html` に上書き。必ず同意を得てから。

---

## 失敗時のリカバリー手順

- **HTMLの見た目がおかしい** → 該当スライドだけ修正。全体を作り直さない
- **構成・ストーリーが違う** → STEP1に戻り構成から再設計
- **pptx変換でエラー** → 原因特定→修正→再実行。2回直らなければ該当要素を簡略化して再試行
- **pptxとHTMLの見た目がズレる** → 座標・色・フォント指定を `design-system.md` の変換対応表に照らして修正
- **どうしても解決できない** → HTMLプレビューを先に納品し、pptxは後から渡す。沈黙しない
